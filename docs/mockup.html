Arquitectura del Sistema

SyWork Tickets es una aplicación interna de gestión de tickets e incidentes para un equipo de consultoría Oracle ERP/CRM. Stack: React + Flask + PostgreSQL on-premise. Metodología: SDD sobre GitHub Spec Kit.

Capas — Clean Architecture

El sistema se divide en tres capas independientes que solo se comunican hacia adentro (dependencia unidireccional):

Capa 3 — Presentación / UI (más externa)

Componentes React, páginas, Kanban, formularios Ant Design. Solo renderiza datos — no contiene lógica de negocio.

↕ llama a services/
Capa 2 — Datos / Infraestructura

Repositorios SQLAlchemy, cliente HTTP para Flask API, módulos de reportes. Implementa las interfaces del dominio.

↕ solo tipos y reglas
Capa 1 — Core / Dominio (más interna)

Lógica de negocio pura: FSM de tickets, reglas SLA, entidades (Ticket, Comment, User). Sin dependencias externas.

Backend — Flask & Swagger

La API sigue el principio API-First: el contrato Swagger se define antes de implementar. Esto permite integrar futuros clientes (Zendesk, Helix BMC, portal de clientes) sin cambiar la API.

GET    /api/tickets              → lista paginada con filtros
POST   /api/tickets              → crear ticket
GET    /api/tickets/{id}         → detalle
PATCH  /api/tickets/{id}/status  → transición de estado (FSM)
POST   /api/tickets/{id}/assign  → asignar resolutor
POST   /api/tickets/{id}/comments → agregar comentario
GET    /api/users/{id}/workload  → carga del resolutor
POST   /api/sessions/start       → iniciar sesión de trabajo
POST   /api/sessions/end         → cerrar sesión de trabajo
El endpoint POST /api/tickets/{id}/assign es intencionalmente un endpoint de backend independiente — no está atado a la pantalla del coordinador. En Fase 2, el AI Dispatcher llamará exactamente este mismo endpoint.

from transitions import Machine

class Ticket:
    states = ['nuevo','asignado','en_progreso','pendiente','resuelto','cerrado']
    transitions = [
        {'trigger':'asignar',  'source':'nuevo',       'dest':'asignado'},
        {'trigger':'iniciar',  'source':'asignado',    'dest':'en_progreso'},
        {'trigger':'pausar',   'source':'en_progreso', 'dest':'pendiente'},
        {'trigger':'reanudar', 'source':'pendiente',   'dest':'en_progreso'},
        {'trigger':'resolver', 'source':'en_progreso', 'dest':'resuelto'},
        {'trigger':'cerrar',   'source':'resuelto',    'dest':'cerrado'},
    ]
    def __init__(self): Machine(model=self, states=self.states,
        transitions=self.transitions, initial='nuevo')

SLA Engine

El motor SLA calcula tiempos respetando el calendario laboral del equipo. Los estados que pausan el SLA son configurables.

  • Estados que descuentan SLA: Nuevo, Asignado, En progreso, Resuelto
  • Estados que pausan SLA: Pendiente (configurable)
  • Los cálculos se basan en el calendario de trabajo (horario, feriados, zona horaria)
  • Alertas automáticas al 50%, 75% y 90% del tiempo consumido
  • Los tiempos se ajustan manualmente post-diagnóstico si es necesario

Frontend — React & Ant Design

Principios de desarrollo frontend según el SDD:

  • Los componentes en /components son "tontos": solo renderizan props, sin lógica de negocio
  • La lógica y llamadas API viven en /services (Principio SRP)
  • Tipado TypeScript estricto — prohibido usar any
  • Archivos de componentes en PascalCase.tsx, utilidades en camelCase.ts
  • DB columns en snake_case

PostgreSQL & Row Level Security

Toda consulta está protegida doblemente: a nivel de API (JWT) y a nivel de base de datos (RLS). Incluso si la API se compromete, un usuario solo puede ver sus propios tickets.

-- Habilitar RLS en la tabla tickets
ALTER TABLE tickets ENABLE ROW LEVEL SECURITY;

-- Política: cada usuario solo ve sus tickets o los asignados a su cliente
CREATE POLICY ticket_isolation ON tickets
  USING (
    assignee_id = current_setting('app.user_id')::uuid
    OR client_id = current_setting('app.client_id')::uuid
  );

Política RLS — aislamiento por tenant

-- Política raíz sobre projects
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
CREATE POLICY project_isolation ON projects
  USING (client_id = current_setting('app.client_id')::uuid);

-- task_lists hereda via JOIN
ALTER TABLE task_lists ENABLE ROW LEVEL SECURITY;
CREATE POLICY tasklist_isolation ON task_lists
  USING (project_id IN (
    SELECT id FROM projects
    WHERE client_id = current_setting('app.client_id')::uuid
  ));

-- tasks hereda via JOIN (incluye subtareas)
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
CREATE POLICY task_isolation ON tasks
  USING (task_list_id IN (
    SELECT tl.id FROM task_lists tl
    JOIN projects p ON p.id = tl.project_id
    WHERE p.client_id = current_setting('app.client_id')::uuid
  ));

Seguridad — JWT & Google OAuth2

  • Login exclusivo por Google OAuth2 restringido al dominio @sywork.net
  • El backend emite un JWT sin estado tras verificar el token de Google
  • El frontend solo almacena el JWT — nunca credenciales del servidor
  • Ningún componente "use client" contiene claves de API o strings de conexión
  • Variables de entorno: .env en backend (nunca comiteado); VITE_API_URL como única variable pública en frontend

AI Dispatcher — Preparación Fase 2

Gold Standard Dataset: Cada asignación que hace el Coordinador humano en Fase 1 se registra como dato de entrenamiento. En Fase 2, el AI Dispatcher aprenderá de estos patrones para asignar tickets automáticamente.

Para que esto funcione, en Fase 1 se requiere:

  • Skills de resolutores estrictamente parametrizadas como etiquetas: JDE_GL, API_REST, Fusion, etc.
  • El endpoint POST /api/tickets/{id}/assign es llamado por el humano hoy, y por la IA mañana — misma función, distinto caller
  • Registrar en DB: quién asignó, a quién, basado en qué skills, con qué carga previa del resolutor
  • El AI Dispatcher futuro será un agente que consume el historial de asignaciones para aprender el patrón óptimo

Metodología SDD & Spec Kit

SyWork Tickets se desarrolla con SDD (Spec-Driven Development) sobre la plataforma GitHub Spec Kit. Este documento explica cómo funciona, qué va en cada archivo y cómo escalar el sistema con nuevas funcionalidades.

¿Qué es SDD?

SDD (Spec-Driven Development) es una metodología donde primero se escribe la especificación completa del sistema y luego esa especificación se convierte en la fuente de verdad que guía tanto al equipo humano como a los agentes de IA en la implementación.

Principio central: "El código sigue a la especificación, no al revés." Ningún componente se implementa sin que exista una spec aprobada que lo describa.

En SDD existen tres documentos vivos que definen todo el proyecto:

📜 Constitución
Reglas inamovibles. El "ADN" del proyecto. Stack, arquitectura, convenciones, seguridad. Nunca cambia sin consenso total del equipo.
📋 Planning
El roadmap. Qué se construye en cada fase, qué dependencias están aprobadas, decisiones de arquitectura con justificación.
🔧 Specs
Una spec por feature. Describe exactamente qué construir, cómo funciona, qué endpoints expone y cuáles son los criterios de aceptación.

Flujo de trabajo SDD

1. Escribir Spec
Se crea o actualiza la spec del feature en docs/specs/ o como GitHub Issue
→
2. Aprobar
El equipo revisa y aprueba la spec. Solo entonces se abre un PR para implementar.
→
3. Implementar
El agente IA o el dev lee la spec aprobada y genera código coherente con la Constitución.
→
4. Validar
Los tests verifican los criterios de aceptación. El PR cierra el Issue de la spec.

Spec Kit — GitHub como plataforma de specs

Spec Kit usa GitHub nativo: Issues, Projects, Labels y Markdown para crear un sistema donde cada feature tiene su especificación versionada y trazable directamente en el repositorio. Los agentes IA como Claude Code leen directamente los Issues para saber qué construir.

  • Cada feature → un GitHub Issue con su spec completa en el cuerpo
  • Labels: spec, implementation, bug, fase-1, fase-2, ai-ready
  • Un GitHub Project (Kanban) rastrea el estado: Spec → Review → In Progress → Done
  • Los Milestones agrupan specs por fase de entrega
  • Los PRs siempre referencian el Issue-spec que implementan: Closes #42

# SPEC: Triage Dashboard
## Feature: Panel de Asignación de Tickets
Fase: 1 | Labels: spec, fase-1, triage | Issue: #7

### Contexto
El Coordinador de Servicio necesita una vista centralizada para analizar
los tickets nuevos y asignarlos al resolutor más idóneo. Para decidir,
necesita ver en tiempo real: skills de cada resolutor y su carga actual.

### Criterios de aceptación
- [ ] Panel muestra todos los resolutores Nivel 2 activos
- [ ] Para cada resolutor: nombre, rol, skills (tags), nº tickets activos
- [ ] Barra de carga visual (verde <30%, amarillo 30-70%, rojo >70%)
- [ ] Resolutor recomendado = skill compatible + menor carga → borde azul
- [ ] Botón "Asignar" ejecuta POST /api/v1/tickets/{id}/assign
- [ ] Al asignar, el ticket transiciona automáticamente nuevo → asignado
- [ ] Coordinador puede enviar mensaje interno al QM desde el panel
- [ ] Los datos de carga se refrescan cada 30 segundos (polling)

### Diseño de API
POST /api/v1/tickets/{id}/assign
Headers: Authorization: Bearer {jwt}
Body:    { "assignee_id": "uuid", "notes": "string (opcional)" }
200 OK:  { "ticket": Ticket, "transition": "nuevo → asignado" }
409:     { "error": "ticket_not_in_new_state" }

GET /api/v1/users/workload
200 OK:  [{ "user_id": "uuid", "name": "Ana P.", "skills": ["JDE_AP"],
             "active_tickets": 2, "workload_pct": 40 }]

### Modelo de datos
assignment_log (nueva tabla para Gold Standard Dataset):
  id, ticket_id, assignee_id, assigned_by, skills_match (bool),
  workload_at_assignment (int), created_at

### Reglas de negocio
- Solo tickets en estado `new` pueden ser asignados desde este panel
- Un ticket solo puede tener un resolutor asignado a la vez
- Si el coordinador reasigna, el estado vuelve a `assigned` (no `in_progress`)
- El log de asignación se registra SIEMPRE para construir el dataset de IA

### Notas técnicas
- El endpoint /assign es independiente de la UI — mismo endpoint que
  usará el AI Dispatcher en Fase 2 (diseño intencional)
- No usar WebSocket en Fase 1 — polling cada 30s es suficiente
- Skill matching: tag del ticket debe existir en el array skills del resolutor

# SPEC: SLA Engine
## Feature: Motor de Control de SLA
Fase: 1 | Labels: spec, fase-1, sla | Issue: #8

### Contexto
Los tickets tienen tiempos máximos de atención, análisis y resolución
según su prioridad. El SLA debe respetar el calendario laboral del equipo
y pausarse cuando el ticket está esperando respuesta del cliente.

### Criterios de aceptación
- [ ] SLA calculado según tabla configurable (prioridad → tiempos)
- [ ] El reloj del SLA respeta el horario laboral (no cuenta fines de semana
      ni fuera del horario configurado en Setup)
- [ ] Los estados pending_user y pending_internal pausan el SLA
- [ ] El sistema registra todos los eventos de pausa/reanudación en sla_events
- [ ] Alertas automáticas al consumir 50%, 75% y 90% del tiempo de SLA
- [ ] El SLA deadline es visible en el detalle del ticket y en el Kanban
- [ ] Los tiempos estimados se actualizan manualmente post-diagnóstico

### Diseño de API
GET  /api/v1/tickets/{id}/sla
200: { "deadline": "ISO8601", "pct_consumed": 74, "paused": false,
       "time_remaining_minutes": 34, "breached": false }

POST /api/v1/tickets/{id}/sla/adjust
Body: { "new_deadline": "ISO8601", "reason": "string" }

### Modelo de datos
sla_events: id, ticket_id, event_type (start|pause|resume|breach),
            state_from, state_to, ts, triggered_by
work_calendar: weekday (0-6), start_time, end_time, timezone
sla_config: priority, attention_min, analysis_min, resolution_min

### Reglas de negocio
- P1 Crítica: atención 30min, análisis 1h, resolución 4h
- P2 Alta:    atención 1h,    análisis 2h, resolución 8h
- P3 Media:   atención 2h,    análisis 4h, resolución 24h
- P4 Baja:    atención 4h,    análisis 8h, resolución 72h
- Los feriados y excepciones se configuran en el módulo Setup
- El campo priority y severity son independientes (ver Constitución §10)

¿Cómo agregar nuevas funcionalidades?

El proceso SDD para agregar cualquier nueva feature —sea pequeña o una fase completa— siempre sigue el mismo flujo. Nunca se empieza por el código.

1
Evaluar si va en una fase nueva o en la fase actual

Si la feature es pequeña y compatible con la Constitución actual → va en la fase en curso. Si requiere nuevas tablas, nuevos roles, integraciones externas o cambia la arquitectura → define una nueva fase en el Planning.

2
Actualizar el Planning

Si la feature necesita nuevas dependencias, agregarlas a la lista aprobada del PLANNING.md con su justificación. Documentar la decisión de arquitectura si aplica.

3
Escribir la Spec

Crear docs/specs/nombre-feature.md con el formato estándar. Abrir un GitHub Issue con label spec y el número de fase. El equipo revisa y aprueba antes de continuar.

4
Implementar con el agente IA

El agente lee la Constitución + la Spec aprobada y genera el código. Cada PR referencia el Issue de la spec. Los criterios de aceptación se convierten en tests.

5
Cerrar el ciclo

Tests pasan → PR aprobado → Issue cerrado → feature en producción. Si algo cambia durante implementación, se actualiza la Spec primero (no el código).

Regla práctica: Si el agente IA te pregunta "¿qué hago ahora?", la respuesta siempre es: "Lee la Spec #X del Issue #Y." Si no hay spec, no hay implementación.

Vitest es el runner recomendado para proyectos Vite/React. Es hasta 10x más rápido que Jest con TypeScript y la configuración es mínima.
